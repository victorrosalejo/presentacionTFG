Metodología — modelo iterativo e incremental
Matiz importante: el nativo ya era un buscador de patrones (TimeBoxes); el TFG lo lleva a otro nivel en dos dimensiones — expresividad y escala. Metodología iterativo-incremental en 4 fases: al evolucionar una herramienta preexistente, los requisitos de interacción no podían fijarse de antemano; cada funcionalidad se probó sobre datos reales antes de la siguiente (un enfoque en cascada habría sido inadecuado). Las fases enlazan con los OE y con la estructura de esta defensa.

Contribución 2 · BVH

Fase amplia — localizar candidatos en O(1)

① El usuario dibuja la consulta
TimeBox, slider, curva o puntos — todo se reduce a un área
② Las celdas salen de una división
col = ⌊ x / cellW ⌋  ·  row = ⌊ y / cellH ⌋
③ Solo esas celdas se inspeccionan
Sus series pasan a la fase estrecha — el resto se ignora
Animación en bucle con 3 fases sincronizadas con los pasos de la izquierda. ① La consulta del usuario se reduce a un bbox. ② Las celdas candidatas salen de dos divisiones enteras (floor) sobre las esquinas min/max — O(1), sin recorrer árboles; alternativas descartadas (quadtree, k-d tree) en el anexo. ③ Solo las series que tocan esas celdas pasan a la fase estrecha; las demás ni se examinan. En el ejemplo: 6 celdas, 2 de 7 series candidatas.
Contribución 2 · BVH

Fase estrecha — 4 primitivas geométricas

① seg – rect
TimeBoxes · fórmula paramétrica
② seg – seg
Curvas de referencia · t, u ∈ [0, 1]
③ ray casting
Corredores above/below · par/impar cruces
④ dist punto–segmento
Tolerancia ε · proyección acotada al segmento
Cuatro primitivas geométricas, cada una diseñada para responder exactamente la pregunta de su tipo de filtro. Las 3 nuevas (seg-seg, ray casting, dist pt-seg) se integran en el BVH con la misma estructura de celdas — solo cambia la función de colisión. Detalle clave: el parámetro t,u ∈ [0,1] garantiza que solo se detectan intersecciones dentro del segmento real, no en la prolongación. Para ε: la proyección paramétrica clampeada mide la distancia perpendicular mínima real al segmento.
Contribución 3 · rendimiento

Canvas 2D — el cuello de botella

① El bucle dibuja serie a serie
for (serie of datos) → ctx.stroke() ← bloqueo
② N bloqueos secuenciales en 1 core
El frame crece linealmente con N — revienta el presupuesto de 16.6 ms (60 fps)
③ N drawcalls → N sincronizaciones CPU↔GPU
Canvas sí usa GPU para rasterizar — el cuello es 1 stroke() por serie: imposible batchear con colores distintos
Bucle de 3 fases. ① El nativo dibuja con Canvas 2D: un for sobre las N series y una llamada ctx.stroke() por serie — cada bloque rojo del timeline es un stroke. ② Esos bloques son secuenciales en un solo core: con N grande el frame rebasa el presupuesto de 16.6 ms (línea roja). ③ Importante matiz: Canvas 2D sí usa la GPU para rasterizar — el procesador gráfico sí trabaja. El verdadero cuello son los N drawcalls independientes: cada ctx.stroke() es una llamada separada al driver con sincronización CPU↔GPU. No se pueden agrupar con Path2D si cada serie necesita su propio color u opacidad. Números (Intel UHD): 70.9 fps N=1k, 16.2 N=5k — inutilizable —, 0.9 N=100k. La degradación es lineal porque el modelo es inherentemente secuencial; hay que cambiar de modelo, no optimizarlo.
Contribución 3 · rendimiento

WebGPU — la migración

① Detección única en init()
navigator.gpu?.requestAdapter()
② Fachada con fallback automático
Con adapter → WebGPURenderer · sin él → Canvas 2D — el cliente no cambia
③ Cada frame, una sola draw call
pass.drawIndexed(totalVerts) ← geometría subida 1×
Misma mecánica de 3 fases que las slides anteriores. ① TimeLineOverview se convierte en fachada: una sola detección al inicializar con navigator.gpu?.requestAdapter() — esa pregunta se hace una vez, no por frame. ② Si hay adapter, instancia WebGPURenderer; si no, cae a Canvas 2D automáticamente. La API es idéntica: TimeWidget.js, el cliente, no cambia ni una línea (rama azul "sí", rama gris "no"). ③ La diferencia estructural: la geometría se sube a la GPU una sola vez al inicio (writeBuffer) y cada frame es una única llamada drawIndexed() — el punto verde que recorre el diagrama es el frame. Frente a las N llamadas stroke() de la slide anterior. Resultado: 16.7 fps frente a 0.9 con N=100k. La ventaja crece con N porque no es una microoptimización: es un cambio de modelo de ejecución.

Pregunta frecuente · rendimiento

¿Por qué no optimizar Canvas 2D?

✓ Canvas 2D sí usa GPU para rasterizar
Cada ctx.stroke() envía geometría a la GPU y ésta la rasteriza. El hardware gráfico sí trabaja.
✕ El cuello: N drawcalls independientes desde CPU
Con N series → N llamadas al driver GPU. Cada una implica un flush de comandos y sincronización CPU↔GPU. El overhead crece linealmente con N.
¿Por qué no batchear con Path2D?
Path2D agrupa trazados pero exige el mismo strokeStyle para todo el batch. Imposible si cada serie necesita su propio color/opacidad para mostrar la selección.
Solución: cambiar de modelo
WebGPU sube la geometría completa una sola vez y aplica estilos por serie mediante un buffer en GPU. Un único draw call — sin N sincronizaciones.
Canvas 2D — N=100 000
for (s of series) {
ctx.strokeStyle = colorDe(s)
ctx.stroke() ← drawcall #N
} × 100 000 sync CPU↔GPU
WebGPU — N=100 000
uploadGeometry(allSeries) // 1× al inicio
styleBuffer.update(colors) // GPU buffer
device.draw() ← 1 drawcall total
16.7 fps (WebGPU) vs 0.9 fps (Canvas 2D) · N=100 000
Respuesta a la pregunta del tribunal sobre reducir los stroke(). Canvas sí usa GPU para rasterizar — lo incorrecto sería decir que la GPU está ociosa. El verdadero cuello es que cada ctx.stroke() es un drawcall independiente al driver con sincronización CPU→GPU. Con N=100k → 100k sincronizaciones. Path2D puede agrupar geometrías pero necesita el mismo strokeStyle para todo el batch; imposible con selección por color/opacidad por serie. OffscreenCanvas desbloquea el hilo UI pero no elimina los N drawcalls. LOD reduce puntos pero no drawcalls. WebGPU cambia el modelo: geometría subida una vez, estilos en buffer GPU, un único draw call. Por eso el factor de mejora crece con N: 1.04× a N=500, 19.18× a N=100k.
